## Окружение

## Архитектура

- Один flexible-content филд `sections`, привязан к главной странице через ACF location rule `page_type == front_page`.
- Каждая секция — отдельный layout внутри `sections`; имя layout = имя файла партиала `template-parts/section/{layout}.php`.
- `front-page.php` крутит `have_rows('sections')` и подключает партиал через `get_row_layout()`.
- ACF-поля — через local JSON (`acf-json/`, фильтры `acf/settings/save_json` и `load_json` в `inc/acf.php`). Группы подхватываются автоматически без похода в БД; «Sync» в админке — финальный шаг импорта.
- Ассеты (`static/css/style.css`, `static/js/app.js`, шрифты, картинки) скопированы как есть в `assets/` темы и подключены через `wp_enqueue_style/script`. SCSS/Pug-исходники дальше не трогаются в теме — правки стилей делаются в `C:\dev\greatjob` и перезаливаются собранные файлы.
- CPT без внутренних страниц (Отзывы, Тарифы) — `public => false`, без archive/single.
- CPT с внутренними страницами (Услуги) — `public => true`, есть `has_archive` и `single-{cpt}.php`; секция на главной и архивная страница переиспользуют один и тот же партиал.
- Header/footer пока не переносятся (используется дефолтный underscores-вывод) — по явному указанию, перенесём отдельно.

## Важный нюанс по image-полям ACF

При тестовой заливке контента через WP-CLI (`update_field`) нельзя передавать в image-поле сырую ссылку на файл — ACF ожидает ID вложения из медиатеки. Сначала нужно реально создать attachment (`wp media import` для PNG/JPG; для SVG — через `wp_insert_attachment` напрямую, т.к. WP по умолчанию блокирует загрузку SVG). Так же нельзя делать `get_field()` → `update_field()` "туда-обратно" для полей с картинками: `get_field()` возвращает уже отформatированное значение (у нас `return_format: url`), и повторное сохранение этой строки вместо ID обнуляет поле. Пользователь при реальном заполнении контента через админку этого не увидит — ACF Pro в UI всегда работает с ID вложений корректно, это специфика только прямых WP-CLI/скриптовых правок.

## SVG-загрузка в медиатеку

WordPress по умолчанию блокирует загрузку `.svg` (защита от XSS). Решено: разрешить — редакторам это понадобится для иконок карточек. Добавлен `inc/svg-upload.php` (`upload_mimes` + `wp_check_filetype_and_ext` фильтры), подключён в `functions.php`. Проверено: `wp media import` для `.svg` теперь проходит стандартным путём без обхода через `wp_insert_attachment`. Санитайзинга содержимого SVG нет — доступ к загрузке есть только у доверенных админов/редакторов сайта, это принято как приемлемый риск для этого проекта.

## Отзывы — детали (первая CPT-секция)

- CPT `greatjob_review`: `inc/cpt/review.php`. `public => false`, без archive/single — как договаривались, карточка отзыва это и есть сам элемент CPT, отдельной страницы нет. `supports: title, page-attributes` — заголовок записи = заголовок карточки (title у review-card), `page-attributes` даёт drag-and-drop сортировку в списке записей (это и есть управление порядком карточек, без ACF relationship-поля).
- ACF field group для CPT: отдельный файл `acf-json/group_greatjob_review_cpt.json` (не в общем `sections`-файле — он про flexible content главной, это про сам CPT). Поля: `body` (текст отзыва), `author` (имя+должность), `rating` (select **0–5**, изначально было только 4/5 — расширено по правке).
- `rating` → звёзды генерируются динамически inline-SVG в партиале, а не подключением готовых файлов `stars-4.svg`/`stars-5.svg`. Разобрал эти SVG: это буквально одна и та же графика 5 звёзд по 24px, у 4-звёздочной последняя звезда просто закрашена серым (`#DDE8EB`) вместо жёлтого (`#FFCA00`). Партиал берёт path одной звезды и рисует 5 штук через `translate(i*24, 0)`, закрашивая первые `rating` золотом — так работает любое значение 0–5, а не только те два, для которых были готовые файлы. Дубликаты `stars-5-alt.svg`/`stars-5-alt2.svg` не используются (экспортные артефакты Figma, визуально идентичны). `author-line-*.svg` — чисто декоративный разделитель, ширина подгонялась в макете под конкретный текст; захардкожен один вариант (`author-line-32.svg`).
- Стрелки навигации слайдера (`reviews__button-icon`) — тоже inline SVG, а не `<img src>`. Причина: у иконок из Figma заливка задана как `fill="var(--fill-0, white)"` — при подключении через `<img>` это в части браузеров не резолвится и иконка становится невидимой (закрашенный SVG теряет fill, наследует `fill="none"` от родителя). Инлайн-SVG с обычным `fill="white"` рендерится предсказуемо везде. Такая же переработка нужна и другим декоративным иконкам, где используется этот var()-паттерн, если где-то ещё вылезет.
- Layout `reviews` в общем `sections`-филде — только заголовок секции (`title`). Сами карточки НЕ хранятся в ACF-поле секции: партиал делает `get_posts()` по `greatjob_review` (`post_status: publish`, `orderby: menu_order`), как договаривались изначально (показываем все опубликованные, без relationship-поля выбора).
- Партиал: `template-parts/section/reviews.php`.
- Проверено: 6 тестовых записей CPT созданы через `wp_insert_post`, все выводятся в правильном порядке; протестирован промежуточный рейтинг (3 из 5 → 3 золотых + 2 серых), `debug.log` чист.

## Header/Footer — детали

- Не flexible-content layout и не CPT — отдельный раздел админки через **ACF Options Pages**: родительский пункт меню «Header & Footer» (`inc/options-pages.php`, `acf_add_options_page`) с двумя вкладками-подстраницами «Header» и «Footer» (`acf_add_options_sub_page`, `redirect: true` у родителя — клик по «Header & Footer» сразу ведёт на вкладку Header). Это именно тот подход, который просили — не нативное меню WordPress (Appearance → Menus), а кастомный экран с ACF-полями. Проверено авторизованным запросом через `wp-login.php` + curl: обе вкладки открываются (`admin.php?page=greatjob-header-options` / `-footer-options`), меню в сайдбаре собирается верно.
- Два field group: `group_greatjob_header_options.json` (location `options_page == greatjob-header-options`) и `group_greatjob_footer_options.json` (location `options_page == greatjob-footer-options`).
- **Подменю**: `nav_items` — repeater с вложенным `sub_items` (тоже repeater). Если у пункта есть вложенные пункты — на десктопе показывается выпадающее меню под родительским пунктом (по наведению/фокусу, чистый CSS, без JS), в мобильном меню — списком под пунктом. В самом Figma-макете подменю не было вообще (плоское меню из 4 ссылок) — это новая функциональность, добавил дизайн сам (радиус/тени/цвета взял из уже существующей палитры компонентов, чтобы не выбивалось).
- `header.php` и `footer.php` темы — полностью переписаны с дефолтного underscores-вывода (`wp_nav_menu`, `bloginfo`) на реальную Figma-вёрстку с данными из Options Pages. Т.к. `header.php`/`footer.php` подключаются на каждой странице сайта (не только на главной), это автоматически закрывает пункт «перенос header/footer под дизайн» — работает и на архиве/страницах услуг тоже.
- Логотип (`brand-mark.svg` в шапке, `brand-logo.png` в футере) остался захардкожен — не ACF-поле, как и декоративная графика в других секциях; отдельно не просили делать его редактируемым.

## Баг: пустые стрелки навигации слайдера (найдено и исправлено)

Реальная причина оказалась не в SVG/var()/inline — это была ложная версия. Настоящая причина: обязательный `style.css` темы (underscores) содержит глобальное `button { padding: 0.6em 1em 0.4em; }`. У `.reviews__button` стоял `font-size: 30px`, поэтому `1em` = 30px → паддинг слева/справа съедал 60px при ширине кнопки 52px (у нас `border-box`) → контенту (иконке) физически не оставалось места, она визуально пропадала, хотя в DOM была. В статике это не проявлялось, т.к. там нет обязательного `style.css` темы и его `button {...}` сброса. Фикс: `padding: 0` на `.reviews__button` + убран ненужный `font-size: 30px` (у кнопки нет текста, только иконка). Это единственный настоящий `<button>`-элемент на сайте (остальные «кнопки» — `<a class="button">`), поэтому баг больше нигде не всплывёт.

## ВАЖНО: ключ "modified" в JSON — обязателен при каждой правке

После первого Sync ACF сравнивает наличие новых изменений в JSON не по содержимому, а по ключу `"modified"` (unix-время) внутри самого файла: `$modified > get_post_modified_time('U', true, $post_id)` (источник: `admin-internal-post-type-list.php` в ACF Pro). У нашего файла изначально не было этого ключа — из-за этого после добавления layout `problems` админка не показывала «Доступна синхронизация», хотя JSON реально отличался от версии в БД. Добавил `"modified": 1784001040` в конец файла (текущий unix-timestamp). **Каждый раз, когда правим `acf-json/*.json` руками — обязательно обновлять `"modified"` на текущее время**, иначе пользователь не увидит кнопку синхронизации в админке.
